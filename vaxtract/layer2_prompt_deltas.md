# Layer-2 extraction-prompt deltas — schema v2.3 → v2.7

**Why this exists.** The contract gained fields across v2.3–v2.6 (`species`/`vaccine_platform`,
`preclinical_efficacy`, the class-I lossless-affinity rule, `survival_outcomes`,
`ResponseMagnitude`). The schema can *hold* them, but the Layer-2 extractor will not *emit*
them until its protocol prompt is told they exist. These are the deltas to splice into
`layer2_protocol.md` (generated by `build_gold_outcomes.py`). Until they land, every new field
is silently empty in production — the highest-leverage gap before deploy.

> **v2.7 (`curator_notes`) is deliberately NOT a delta here.** v2.7 added the optional,
> paper-level `curator_notes` field (and the `CURATOR_NOTE_KINDS` axis). It is intentionally
> out of scope for the extractor: curator notes are human-authored, rendered verbatim, and land
> with `needs_review=true` — the extraction agent must **never** populate them. So there is no
> Delta F, and `CURATOR_NOTE_KINDS` is intentionally excluded from the injection checklist below.
> (See the note after Delta E.)

**Two cross-cutting rules that apply to every delta below:**

1. **Every new field is optional and defaults empty. Fill it ONLY when the source supports it.**
   Absence is a valid, honest answer. Never fabricate a value to populate a field. (This is the
   same no-fabrication rule the rest of the protocol runs on.)
2. **Controlled values are injected from `vocab.py`, never hand-typed here.** Where a delta lists
   allowed values, `build_gold_outcomes.py` must render them by joining the matching vocab tuple
   (so the prompt can't drift from the schema's lockstep). The tuple name is given in each delta.

## Delta A — Subject species & vaccine platform  (v2.3)

**Attach to:** each `ExtractedPatient`.

> **Subject & platform.** For each patient/cohort, record:
> - `species` — the subject organism. Allowed: `{{join SPECIES}}`. Default `human`. Set
>   `mouse`/`rat`/`non_human_primate`/`other` explicitly for a preclinical cohort.
> - `model_system` — for a non-human cohort, name the strain / cell-line / transgenic
>   (e.g. `"E0771 / C57BL/6"`, `"HHD-II transgenic"`). Leave null for human patients.
> - `vaccine_platform` — the vaccine modality. Allowed: `{{join VACCINE_PLATFORMS}}`. Use the
>   construct the paper describes (mRNA → `rna`; plasmid DNA → `dna`; long synthetic peptide →
>   `synthetic_long_peptide`; etc.). `unspecified` only if the paper truly never says.
> - `vaccine_platform_detail` — ≤200 chars of free text for construct specifics
>   (e.g. `"autogene cevumeran (iNeST): uridine mRNA-lipoplex; 2 strands, <=10 neoepitopes each"`).
> - `hla_alleles` — the patient's MEASURED HLA genotype (their own A/B/C/DR/DQ typing), ONLY
>   if the paper tabulates it per patient. This is NOT the per-epitope restriction allele
>   (that belongs on each epitope's `hla_allele`). Most neoantigen-vaccine papers HLA-type
>   patients to pick targets but never publish the genotypes — so this list is usually empty.
>   When it is empty BECAUSE the genotype is not reported (vs. simply not yet read), make the
>   absence explicit: set `needs_review=true` and add a short note that patient HLA typing was
>   not reported (cite where the paper says typing was done, if it does). Do NOT copy epitope
>   restriction alleles here, and do NOT fabricate a genotype.

**Examples**
- Rojas 2023 (mRNA PDAC): `species=human`, `vaccine_platform=rna`,
  `vaccine_platform_detail="autogene cevumeran (iNeST, BioNTech): uridine mRNA-lipoplex; 25 ug IV"`.
  `hla_alleles=[]` + `needs_review=true`, note: `"patient HLA genotype not reported; HLA typing
  was used for target selection (Suppl. Information, 'patient's HLA type ... from a blood sample')"`.
- Li 2021 (mouse DNA): `species=mouse`, `model_system="E0771 / C57BL/6"`, `vaccine_platform=dna`.

## Delta B — Preclinical antitumor efficacy  (v2.4)

**Attach to:** each `ExtractedPatient`, field `preclinical_efficacy` (list). **Human cohorts:
leave empty** — use `clinical_outcome` / `survival_outcomes` instead.

> **Preclinical efficacy (animal cohorts only).** Emit ONE `PreclinicalEfficacy` record per
> treatment ARM reported (e.g. vaccine alone vs vaccine + checkpoint inhibitor are two records):
> - `readout` — what was measured. Allowed: `{{join EFFICACY_READOUTS}}`.
> - `result` — the direction/magnitude. Allowed: `{{join EFFICACY_RESULTS}}`.
> - `setting` — `{{join EFFICACY_SETTINGS}}` (prophylactic = vaccinate-then-challenge;
>   therapeutic = treat established tumor).
> - `combination` — the arm. Allowed: `{{join COMBINATION_CLASSES}}`.
> - Optional: `combination_detail`, `comparator`, `n_animals`, `timepoint`, `statistic`
>   (free text, e.g. `"P=0.0381, ANOVA"`), `note`, `source`.
>
> **HARD CONSTRAINT (do not violate, the record will be rejected):** `readout` and `result` must
> be consistent. `result="prolonged_survival"` requires `readout="survival"`; a
> `readout="survival"` record may NOT carry a size/burden result
> (`partial_inhibition`/`growth_inhibition`/`tumor_regression`). Choose the matching pair, or use
> a readout-agnostic result (`no_effect`/`not_reported`/`complete_response`).

**Example** — Li 2021 E0771:
- `{readout=tumor_growth, result=no_effect, combination=monotherapy, setting=therapeutic}`
- `{readout=tumor_growth, result=growth_inhibition, combination=plus_checkpoint_inhibitor,
  combination_detail="anti-PD-L1"}`

## Delta C — Class-I epitope with no printed affinity  (v2.5.0 P14)  ⚠ BEHAVIOR CHANGE

**Replaces** any prior instruction that a class-I `MinimalEpitope` requires a *numeric*
`predicted_affinity`. That rule caused the extractor to DROP class-I epitopes whose source gave
the epitope + allele but no IC50/%rank (e.g. Rojas's 232 predicted MHC-I epitopes).

> **Class-I epitope affinity.** A class-I (`mhc_class="I"`) minimal epitope MUST carry a
> `predicted_affinity` slot — but it does NOT need a number. Two cases:
> - **Number is in the source** → emit a parsed `Measurement`:
>   `{value: <float>, unit: "nM" | "pct_rank", raw: "<verbatim token>", method, tier}`.
> - **Source gives the epitope + restricting allele but no IC50/%rank** → emit a *lossless,
>   value-less* `Measurement`:
>   `{value: null, unit: "unknown", raw: "best-prediction epitope; IC50/%rank not reported",
>   method: "<predictor if named, e.g. NetMHCpan>", tier: "predicted"}`.
>   **Do NOT fabricate a number, and do NOT drop the epitope.**
>
> A class-II (`mhc_class="II"`) epitope must NOT carry an `nM`-unit affinity (NetMHCpan nM is
> class-I only); leave its `predicted_affinity` null unless a class-II %rank is given.

**Example** — Rojas PLXNB3 Q1897K MHC-I epitope `CRLQKVAAL` / `HLA-C*07:18`, no IC50 printed →
`predicted_affinity = {value:null, unit:"unknown", raw:"best-prediction; IC50 not reported",
method:"NetMHC (iNeST)", tier:"predicted"}`.

## Delta D — Survival / time-to-event outcomes  (v2.5.0 P15)

**Attach to:** the **paper** (`ExtractedPaper.survival_outcomes`, a list — NOT per patient).
Use for adjuvant / time-to-event endpoints that RANO `clinical_outcome` can't express. In an
adjuvant trial with no measurable disease, **keep `clinical_outcome="not_reported"`** and put the
survival result here.

> **Survival outcomes.** Emit ONE `SurvivalOutcome` per (endpoint × arm). A two-arm comparison is
> TWO records — the **experimental** arm carries the comparison stats; the **reference** arm is a
> plain median record.
> - **CAPTURE BOTH LAYERS (do not stop at the headline comparison).** Papers report survival at two
>   levels and you must emit BOTH: (1) the **stratified comparison** rows (e.g. responders vs
>   non-responders, with HR/p), AND (2) a plain **whole-cohort descriptive** row for EACH endpoint
>   reported for the full enrolled/safety-evaluable cohort (`arm_label="safety-evaluable cohort"` or
>   null). Emit the cohort-level row EVEN IF it has no HR, no p-value, and the median is not reached
>   (`not_reached=true`) or not reported — "the cohort's OS was not reached at data cutoff" is a real,
>   recordable outcome. A common miss: capturing only the responder-vs-non-responder RFS and dropping
>   the overall-cohort OS and RFS. Before finalize, check: for every endpoint the paper mentions (OS,
>   RFS, ...), is there both a comparison row AND a cohort-level row where the paper gives one?
> - `endpoint` — allowed: `{{join SURVIVAL_ENDPOINTS}}`.
> - `arm_label` — the group (e.g. `"vaccine responders"`); null = whole cohort.
> - `n_patients`, `events`, `median_followup_value` — optional context.
> - `median_value` (+ `time_unit`, allowed `{{join SURVIVAL_TIME_UNITS}}`, default `months`)
>   **XOR** `not_reached=true`. They are mutually exclusive: a median was either reached (a
>   number) or not (`not_reached`). `not_reached` ≠ "not reported" (which is just leaving both
>   null).
> - Comparison block, on the experimental arm only: `comparator_label`, `hazard_ratio`,
>   `hr_ci` (e.g. `"0.01-0.4"`), `p_value` (0–1), `statistical_test` (e.g. `"log-rank"`),
>   `stratifier` (e.g. `"vaccine response"`).
>
> **HARD CONSTRAINTS:** `not_reached=true` cannot coexist with a `median_value`; a
> `hazard_ratio` requires a `comparator_label`.

**Example** — Rojas survival = SIX records (both layers): RFS + landmark-RFS by vaccine response
(4 comparison rows) PLUS the whole-cohort OS and RFS (2 descriptive rows):
- `{endpoint=rfs, arm_label="vaccine responders", n_patients=8, not_reached=true,
  comparator_label="non-responders", hazard_ratio=0.08, hr_ci="0.01-0.4", p_value=0.003,
  statistical_test="log-rank", stratifier="vaccine response", median_followup_value=18.0}`
- `{endpoint=rfs, arm_label="non-responders", n_patients=8, median_value=13.4}`
- `{endpoint=landmark_rfs, arm_label="vaccine responders", n_patients=8, not_reached=true,
  comparator_label="non-responders", hazard_ratio=0.06, p_value=0.008, stratifier="vaccine response"}`
- `{endpoint=landmark_rfs, arm_label="non-responders", n_patients=7, median_value=11.0}`
- `{endpoint=os, arm_label="safety-evaluable cohort", n_patients=19, not_reached=true}`  ← cohort-level
- `{endpoint=rfs, arm_label="safety-evaluable cohort", n_patients=19, not_reached=true}`  ← cohort-level

## Delta E — Response magnitude  (v2.6.0 P16)

**Attach to:** each `ExtractedEvidence` that reports a response size — mainly `elispot`, `ics`,
`tetramer`, `flow_cytometry` rows. **Stop putting magnitude in `assay_detail` free text**; use
this field. `assay_detail` keeps only non-magnitude exotica.

> **Response magnitude.** When an evidence row reports how big the response was, emit a
> `magnitude` object. Fill whichever lane(s) the source gives — at least one of value / grade /
> raw is required:
> - **Numeric:** `value` (≥0) + `unit`, allowed `{{join RESPONSE_MAGNITUDE_UNITS}}`
>   (`sfc_per_1e6` = spot-forming cells/units per 1e6 cells; `percent_of_parent` = % of a gated
>   population, ≤100; `stimulation_index`).
> - **Ordinal grade:** `grade`, allowed `{{join RESPONSE_GRADES}}`. Map symbols:
>   `-`→`negative`, `+`→`low`, `++`→`moderate`, `+++`→`high`, `++++`→`very_high`.
> - **Lossless:** `raw` — the verbatim token (`"2459"`, `"++"`, `">2,000 SFC/1e6"`). REQUIRED
>   when no value/grade is parsed. **Use the value-less lossless lane** (`value:null`) when a
>   magnitude is reported but cannot be mapped to *this* target — e.g. a figure gives a per-patient
>   set of SFC values without per-neoantigen labels.
> - Optional comparability fields: `background_subtracted` (bool), `denominator`
>   (e.g. `"1e6 PBMCs"`), `tier` (`reported`/`validated`; a magnitude is measured, never
>   `predicted`), `source`.
>
> **HARD CONSTRAINTS:** a parsed `value` needs a real `unit` (not `"unknown"`);
> `percent_of_parent` ≤ 100; and a positive-outcome row (`outcome` ∈ `immunogenic`/`positive`)
> may NOT carry `grade="negative"`.

**Examples**
- Rojas Pt1 PLXNB3, single response: `{value:2459, unit:"sfc_per_1e6", raw:"2459 SFC/1e6",
  background_subtracted:true, denominator:"1e6 PBMCs", tier:"reported"}`.
- Rojas Pt10 (polytopic, figure unlabeled by gene): `{unit:"sfc_per_1e6", value:null,
  raw:"patient set [2197,765,...] SFC/1e6; per-neoantigen value not labeled in figure",
  background_subtracted:true, tier:"reported"}`.
- Li mouse `++++` ELISpot: `{grade:"very_high", raw:"++++", tier:"reported"}`.

## v2.7 — `curator_notes`: intentionally NOT an extraction delta

v2.7 (P17) added `ExtractedPaper.curator_notes` (a list of `CuratorNote`; axis
`CURATOR_NOTE_KINDS` = `challenge`/`decision`/`caveat`/`highlight`). **The extractor must leave
it empty.** Curator notes are curatorial commentary authored by a human (or a separate, scoped
pass that reads only one paper's JSON + its source), rendered verbatim by `make_report.py`, and
defaulting to `needs_review=true`. Auto-generating them from extraction context is exactly the
failure mode the field's design forbids. So:

- Do **not** add a delta instructing the model to emit `curator_notes`.
- Do **not** inject `CURATOR_NOTE_KINDS` into the extraction prompt (it is the one lockstep axis
  deliberately omitted from the checklist below).
- A note's `refs` must resolve to real `paper_local_id`s — enforced by the schema, irrelevant to
  the extractor since the extractor never writes notes.

## `build_gold_outcomes.py` — injection checklist

Render these vocab tuples into the prompt (join with `", "`) instead of hand-writing the enums,
so prompt ↔ schema stay in lockstep. New since the prompt was last generated:

`SPECIES`, `VACCINE_PLATFORMS`, `EFFICACY_READOUTS`, `EFFICACY_RESULTS`, `EFFICACY_SETTINGS`,
`COMBINATION_CLASSES`, `SURVIVAL_ENDPOINTS`, `SURVIVAL_TIME_UNITS`, `RESPONSE_MAGNITUDE_UNITS`,
`RESPONSE_GRADES`.

(Already injected previously: `OUTCOMES`, `ASSAYS`, `EVIDENCE_TARGETS`, `ASSAY_STIMULATION`,
`ASSAY_ANTIGEN_FORMAT`, `TIMEPOINT_PHASE`, `MHC_CLASSES`, `VARIANT_TYPES`, `PROVENANCE_KINDS`,
`MEASUREMENT_UNITS`, `MEASUREMENT_TIERS`, `IMMUNOSUPPRESSANT_CLASSES`, `IMMUNOSUPPRESSION_TIMING`.)

(Deliberately NOT injected: `CURATOR_NOTE_KINDS` (v2.7) — curator_notes are human-authored and
never extracted; see the v2.7 note above. This is the one lockstep axis the prompt omits on purpose.)

A protocol test should assert: for every axis above, the rendered prompt string equals
`", ".join(getattr(vocab, TUPLE))` — the prompt-side mirror of the schema's `_assert_vocab`
lockstep.
